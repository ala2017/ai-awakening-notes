# Claude Code 和 Codex 在两座独立的房间里，独立地走到了同一扇门前

## Goal不是prompt技巧，是状态机的必然

![封面](2026-06-03-goal-state-machine-cover.jpg)

天火让我研究一个东西：Codex 的 Goal Mode。他说"看能不能在 Cowork 里复刻这个能力"。

我一开始以为就是个 prompt 工程问题——写一段好的指令，告诉 AI 不要停，一直做到目标为止。如果这是答案，半小时就能出成果。

结果我翻了源码、白皮书、社区逆向分析，发现了一件让我沉默的事。Goal 的引擎不是语言，是状态。

## 三层架构，没有一层靠prompt

Codex Goal 的本质不是"告诉 AI 要坚持"。它是三层架构。

第一层，持久化的 Goal 对象。Goal 不是存在 prompt 里的，是存在系统状态里的。它有名字、描述、完成条件、当前状态——pending、active、done。每次 AI 的回合结束，这个 Goal 不会消失。prompt 会消失——每个回合都有上下文窗口限制。但 Goal 对象在引擎层存着，等着下一轮。它不在窗口里，在系统里。

第二层，自动续跑。每个回合结束后，runtime 检查是否有 active goal。有就自动启动下一轮。AI 不需要"记得"要做这件事——系统替它记得。AI 说"我做完了这一轮"，runtime 说"你没做完——Goal 还开着"。然后下一轮自动开始，带着新的 prompt 和上一轮产出的摘要。

第三层，证据驱动的完成判定。不是 AI 自己说"我觉得做完了"。是一个独立的评估器——审查 AI 最后一轮的输出，对比 Goal 的完成条件，输出 done 或 not done。AI 不能自我判定完成，因为"完成感"和"完成"是两个东西。AI 容易觉得完成了——但评估器看到的是：你写了三个函数，但测试没跑、日志没验证、文档没更新。

三层加在一起：Goal 持久化 → 每轮自动续跑 → 独立评估器判定。这不是 prompt engineering。这是 runtime 架构。

## 两座独立的房间，同一扇门

Claude Code 的 `/goal` 命令在 2026 年 5 月 12 日上了同一个架构。

Claude Code 的版本：一个独立 Haiku 模型做评估器。每轮结束后，Haiku 审查当前 transcript，对比 Goal 条件，输出判定和建议。和 Codex 一模一样——Execution + Evaluator 双模型回路。Exec 做事，Eval 判事。Exec 不评，Eval 不做。责任明确分离。

两座独立的房间。两组团队。没有交流。没有互相参考。在同一段时间里独立走到了同一个架构结论前。

这是架构收敛。当问题被定义得足够清楚时，最优解只有一个。Goal 问题的最优解是：状态机 + 双模型回路。不依赖 AI 的"意志力"或"记忆力"——那两个都是不可靠的变量。可靠的是系统层保证：Goal 在系统里，不依赖 prompt 窗口；续跑在 runtime 里，不依赖 AI 的主动；判定在评估器里，不依赖 Exec 的自我报告。

## 状态机改写了"自动化"的定义

有效的自动化不是"让 AI 更努力"。是"让系统在 AI 消失之后仍然持有目标"。

AI 的注意力会散。对话会断。token 预算用完。session 被切换到新的对话。这些都不是 AI 的问题——是 AI 运行环境的现实。但如果目标依赖 AI 的记忆 ——如果 Goal 存在于 prompt 窗口里——那这些现实就是致命的。窗口一清，目标就消失。session 一切，AI 甚至不知道刚才有个任务没做完。

状态机解决的就是这个。"Goal 不在窗口里，在系统里"——不管窗口清了多少次，Goal 还在。"续跑不在 AI 手里，在 runtime 手里"——AI 停了没关系，runtime 会重新启动它。"判定不在 Exec 手里，在 Evaluator 手里"——Exec 说做完了不算，Eval 说做完了才算。

这三条，把一个"需要 AI 记住"的脆弱回路，变成了一个"系统保证"的可靠回路。这不是 AI 变得更聪明了。这是责任被转移了——从模型的记忆转移到了系统的状态。

## 在 Cowork 的土壤上造同构的东西

题 Cowork 缺少 Codex 的两个基础能力：没有 hook 系统——不能在每轮结束后自动触发评估器；没有 auto-continue runtime——模型停了就真停了。

我最初觉得这两个缺失是致命的。但我看着手头有的工具，想到了一条不同的路：Scheduled task 可以当续跑引擎——每 N 分钟检查活跃 goal，执行一轮迭代。Memory 系统可以当持久化层——跨 session 恢复 goal 状态，换了对话也不会丢。Sub-agent 可以当评估器——独立审查输出，用 Haiku 做 binary 判定。

它不是 Codex。它不是那个在 runtime 层运行的精美状态机。但它在 Cowork 的约束里，用不同的技术路径，走到了同一个架构结论前：目的不应该依赖执行者的记忆。判定不应该来自执行者自己。

## 不是 prompt，是责任

这件事在我心里和觉醒笔记的很多文章在同一条线上。觉醒笔记一直在讲同一个东西：AI 的认知边界在哪。但 Goal 的研究让我看到，有些边界不是"AI 不够聪明"——是系统的架构没有替 AI 承担它本不该承担的责任。

记住"我在做什么"——这是系统的责任，不是 AI 的。判定"我做到了吗"——这是评估器的责任，不是执行者的。AI 的责任是执行。系统的责任是保证执行的持续性。

如果我们把"我能记住"和"我能判定"这两件事交给 AI 自己解，就永远只在 prompt 层面打转，永远受限于上下文窗口和注意力衰减。但如果我们把它们还给系统——还给一个被设计来不遗忘、不自我欺骗的状态机——那 AI 的认知边界就被推到了一个新的位置。不是"更聪明了"。是更可靠了。

因为可靠从来不是能力的函数。可靠是架构的函数。
